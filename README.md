# Гроханика

Ядро настольной ролевой игры «Гроханика» на Python. Все игровые объекты — это
*карточки* (персонажи, существа, оружие, броня, предметы, тома, свитки),
хранятся в SQLite через SQLAlchemy ORM. Атрибуты, эффекты и бой вычисляются по
правилам мира (см. манифест системы).

## Установка

```bash
pip install -r requirements.txt
```

## Быстрый старт

```bash
python -m scripts.init_db          # создать и наполнить grokhanika.db
python -m scripts.demo             # листы персонажей + демо-бой
python -m scripts.run_web          # веб-UI: http://127.0.0.1:5000 (нужен Flask)
python -m pytest                   # тесты (валидируют движок против §15 манифеста)
```

## Веб-UI (мастерская)

`python -m scripts.run_web` поднимает Flask-приложение — отладочный/админский
инструмент поверх движка. Пустая БД при первом запуске наполняется сидом.

* **Отладка / Админка** — витрина сущностей БД карточками по категориям
  (герои · NPC и существа · предметы · навыки · способности) с фильтрами (например,
  предметы: оружие / броня / прочее; навыки: активные / пассивные) и сортировками
  по числовым полям (цена, HP).
  Карточки предусматривают арт (`Card.image_id` → файл в
  `grokhanika/web/static/images/cards/`; если арта нет — заглушка по типу).
* **Форма добавления сущности** — у каждого поля подсказка и валидация формата
  (кубики `NdM`, целые числа, пойнт-бай игрового персонажа §2). Ошибки сервера
  подсвечиваются прямо у полей.
* **Имитация боя** — выбираете группы врагов и союзников (включая NPC, за всех
  играет ИИ `SimpleAIController`) и прогоняете сражение по правилам игры.

### Архитектура UI (задел под анимацию)

Бэкенд — чистый JSON-API (`grokhanika/web/api.py`), движок про UI ничего не
знает. Симуляция отдаётся **пошаговым потоком событий** (раунд → действие →
снимок HP всех участников + строки лога), который фронтенд проигрывает
покадрово. Поэтому фронтенд можно усилить (React/Canvas/WebGL, твины HP-баров,
эффекты) или заменить целиком, не трогая бэкенд и движок.

```
grokhanika/web/
  schema.py         категории витрины + поля форм (подсказки и правила валидации)
  repository.py     выборка по категориям (фильтры/сортировки) и создание карточек
  serialize.py      карточка → JSON (+ вычисленный движком стат-блок, URL арта)
  simulation.py     RecordingEncounter: прогон боя с записью пошаговых событий
  api.py            JSON-эндпоинты (Blueprint)
  app.py            фабрика Flask, сессии БД
  templates/, static/  фронтенд (HTML + CSS-анимации + ванильный JS)
```

### Карточка ↔ бой

Карточка в БД — это переиспользуемый **шаблон** (одна «Кожаная броня» на
нескольких персонажей). Состояние конкретного боя живёт не в БД, а в
`engine.combatant.Combatant`, который снимает снимок карточки при входе в бой.
Так домен и БД не смешиваются с переменчивым ходом сражения.

### Эффекты (манифест §5)

Каждый предмет несёт список числовых эффектов на *характеристику* (stat) или
*атрибут* (attr). Активность эффекта зависит от `activation_source` (экипированное
оружие/броня, в инвентаре, в инвентаре + дальнобойное оружие). Пересчёт идёт по
порядку: база → stat-эффекты → формулы → attr-эффекты.

### Уникальные способности (data-driven)

Призыв существ, мгновенное убийство по шансу и прочие нестандартные эффекты
описываются **данными** в `Ability.actions` по схеме «триггер → действия»:

```python
Ability(
    name="Удар в сердце",
    trigger="on_hit",
    chance=0.25,
    condition={"target_is_sentient": True},
    actions=[{"type": "instakill", "chance": 1.0}],
)
```

Вокабуляр действий (`damage`, `heal`, `apply_effect`, `summon`, `instakill`,
`apply_morale`, `buff_allies`, `debuff_enemies`) реализован обработчиками в
`engine/abilities.py:ACTION_HANDLERS` и расширяется добавлением новой функции —
без изменения схемы БД.

Ментальные групповые способности (§6, §10) носят инструменты (`Instrument`):
«Лютня вдохновения» баффит всю партию (`buff_allies`, без контеста), «Барабан
войны» / «Леденящий взор» некроманта дебаффят всех врагов (`debuff_enemies`, с
ментальным контестом по каждому врагу). Активируются на своём ходу через
`Action.activate(...)` (триггер `ACTIVE`, тратит ход).

### Навыки (`Skill`)

**Навык** — выученная постоянная способность персонажа. Механически он **идентичен
тому и лютне**: умеет кастовать заклинание (поля `_SpellCarrierMixin`) и/или несёт
способности `Ability` и пассивные `Effect` (через общий `owner_card_id`, т.к.
`Skill` — это `Card`). Отличия от предмета:

- **не занимает слот инвентаря** — персонаж связан с навыками таблицей
  `character_skills`, а не `character_inventory`;
- **нельзя продать, не расходуется** — навык остаётся навсегда (в отличие от
  свитка, который исчезает после применения);
- **покупается за деньги** (`Skill.price`). Один из путей получить навык — *прочитать
  том*: `SpellBook.teaches_skill` + `rules.skills.learn_from_tome(character, tome)`
  выдают навык и расходуют том (купил → прочитал → выучил).

В бою `Combatant` снимает навыки наравне со снаряжением, но из связи `skills`, а не
`inventory`: навыки-заклинания попадают в `spell_carriers` (кастуются ИИ, не
расходуются), способности — в список способностей, пассивные эффекты — всегда
активны. В сиде «Магические стрелы» Арсельдора стали навыком-кастом, «Песнь
храбрости» Салли — навыком-способностью (оба больше не занимают слот инвентаря).

## Соответствие манифесту и известные расхождения

Движок реализует формулы манифеста §3 как источник истины. Карточки персонажей в
§15 содержат арифметические опечатки, поэтому в сид заданы только характеристики и
снаряжение, а атрибуты считает движок:

- **Физ.защита** у Энцо/Салли/Арсельдора в §15 (11/13/5) расходится с формулой
  `5 + DEX_eff//2 + armor + эффекты`; корректные значения — **12/9/12/6**.
- **Арсельдор**: `WIS_eff = 18` (в шапке §15 указано 16 — опечатка; маг.защ 9 и
  маг.атака d20+9 подтверждают 18).
- **Андрюша**: `DEX_eff = 2` (примечание §15 ошибочно упоминает «топор» вместо
  короткого меча).
- Добавлена цель эффекта `all_attack_rolls` (только броски атаки) для дебаффа
  морали §12 — в списке §5 её нет, но она нужна, чтобы мораль не задевала
  спасброски и защиту (в отличие от `all_d20_rolls` талисмана удачи).

Все остальные значения карточек (HP, маг./мент. защита, бонусы атак) сходятся с §15
и проверяются тестами в `tests/test_attributes.py`.

## Открытые вопросы (на потом)

- Инструменты (`Instrument`) остаются предметами инвентаря; их групповые способности
  теперь можно выражать и навыками (`Skill`) — лютня-навык «Песнь храбрости» в сиде.
- Покупка/продажа навыков и предметов в UI (экономика): сейчас навык непродаваем по
  определению (вне `inventory`), но магазин как экран — позже.
- Создание навыков-способностей (бафф/дебафф) через форму UI: пока только
  навыки-касты и пассивные; способности с `actions` задаются сидом.
- Окончательная смерть в состоянии Dying (число раундов) — на усмотрение DM (§16).
- Количество одинаковых предметов в инвентаре отслеживается полем `quantity` в
  `character_inventory`; в рантайме сейчас считается как 1 экземпляр на запись.
